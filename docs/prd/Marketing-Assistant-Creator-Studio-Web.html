01 — Purpose & Goals

Close the loop: creators can build a course; they still can’t sell one.

WiseSel’s studio turns expertise into a course. But a finished course earns nothing until someone finds it, trusts it, and buys it — and that is exactly where solo creators stall. The Marketing Assistant is the second half of the product: it generates the go-to-market assets from the course itself, captures leads, runs the email lifecycle on our own state machine, measures everything through one event stream, and lets an agent operate all of it behind a reversibility-graded approval gate. The same conviction as the Content Agent — ambitious in the AI, conservative in what reaches a real person.

The problem

  • Marketing is a different skill than teaching. Creators have a finished course and no landing page, no list, and no launch plan.
  • The tooling is fragmented and disconnected. A page builder, an email tool, and an analytics tool — none of which know what the course actually teaches, so every asset is written from scratch.
  • Generic “AI marketing” writes copy but doesn’t act or learn. It can’t publish a page, schedule a sequence, watch the funnel, and propose the next move — and a creator won’t let an autonomous tool email their list without a hand on the brake.

Goals

One engine, grounded in the course

A landing page, launch sequence, and followups generated from the syllabus, outcomes, audience, and tone — not a blank template. The course is the source of truth.

Trust by reversibility

Every mutation routes one governance gate. Drafts auto-stage and can be rejected atomically; anything outward-facing (publish, send) waits for explicit human approval — whether the actor is the agent, a card, or a button.

One architecture, three surfaces

A single typed tool layer behind the “Generate Kit” button, the tool cards, and the agent chat. Built mock-first so the whole loop works before Resend exists, then real integrations swap in with zero contract changes.

Non-goals (this milestone)

  • No social posting and no per-creator email OAuth — sends go out on the platform’s own verified Resend domain; per-creator domains are a later seam.
  • No paid-ads management, no Stripe/checkout changes, no marketplace changes. “Enrollment” is recorded as an event; payment lives elsewhere.
  • No A/B testing, no segmentation builder beyond saved segments, no CRM/pipeline.
  • No image generation for pages — leave a clean seam (placeholder/library images only).

Hard non-negotiables (enforced by design)
  • The OpenAI key, the Resend key, and every model/email call are server-side only; each provider SDK is imported in exactly one file.
  • Every mutating tool call routes the governance gate — agent, tool card, or batch button alike. There is no second write path, and even the mock “send” passes the gate.
  • Irreversible actions require explicit human approval (publish live, send to a list, activate a sequence, bulk-enroll). Reversible actions auto-stage and are rejectable with atomic before-snapshot rollback.
  • One event stream (analytics_event) renders the dashboard and feeds the agent’s observe step; subscriber status is a reducer over it, never a parallel log.
  • All creator data is author-scoped by RLS. Public lead/analytics writes go through a controlled server ingest path, never anonymous table writes.
  • Every marketing email carries consent provenance and a working one-click unsubscribe; migrations & RLS require human sign-off before they are applied.
Decisions for sign-off (defaults chosen — flag any to change)
  • Public ingest path → service-role server route default. A server-only SUPABASE_SERVICE_ROLE_KEY ingest endpoint validates the target page is published, then writes leads/events — instead of opening anonymous INSERT RLS. Keeps validation, dedup, and bot-filtering in one place.
  • One campaign per course for MVP default. The schema already allows many; the UI assumes one to keep the model simple.
  • send_test_email is gated like any send default. It only ever targets the creator’s own verified address, but it still routes the gate so “every send is gated” has no exceptions.

03 — Product Approach

One typed tool layer, one event stream, one reversibility-graded gate.

Approaches considered

A · Embed a third-party suite

Wire up Kit / Mailchimp via API for email + pages.

Rejected: no course grounding, no agent, and the creator’s list + analytics live outside the product. Fragments the very data the agent needs to observe.

B · Shared tool layer + gate + agent, mock-first ✓

One typed tool contract behind every surface; a reversibility-graded gate on all mutations; a reason→act→observe agent; built against mock providers, real ones swap in by env.

Recommended: grounded, trustworthy, autonomous-capable, and de-risked — the whole loop runs before any external integration exists.

C · Standalone template gallery + manual editor

A page/email builder with manual editing only.

Deferred: no leverage and no agent — and it’s simply the renderer that already sits beneath B. We build it as B’s substrate, not as the product.

The governance gate — reversibility decides everything

The gate is a single function every mutating tool call passes through, regardless of origin. It reads the tool’s declared reversibility grade and routes accordingly. This is the unification of the studio’s Accept/Reject change-set and an approval requirement into one first-class primitive, recorded in the marketing_action ledger.

Tool callReversibilityGate behaviorLedger status
readn/a (no mutation)Executes immediately; never staged.not recorded
generate / edit draftReversibleExecutes immediately, captures a before_snapshot, and stages the result as rejectable. Reject replays the snapshot atomically.auto_approved → executed (or reverted)
publish / send / activateIrreversibleDoes not execute. Logs a pending action, surfaces an approval request, and pauses the agent loop. Human approve executes it; deny discards it.pending → approved/executed (or rejected)

Why one gate for all three surfaces: because the human clicking “Send” in a tool card routes the same gate() as the agent calling send_broadcast, the safety property holds no matter who acts. The autonomy boundary is the gate: the agent chains reversible steps freely, but must stop at every irreversible one.

Key business rules

Server-only provider accessThe agent loop, sends, and ingestion run in route handlers. The browser never sees the OpenAI key, the Resend key, or raw provider traffic.
Every mutation routes the gateNo tool writes outside gate(). Reversibility is a property of the tool contract, not a UI decision; even the mock send is gated.
Our scheduler, not Resend’sSequence/followup orchestration is our state machine on our outbox. Resend (or the mock) only moves bytes; it never owns sequence logic or timing.
Idempotent sendsA unique (touch_id, subscriber_id) on the outbox guarantees a subscriber is never sent the same touch twice, even if the scheduler ticks repeatedly.
One event streamAll measurement is analytics_event rows. The dashboard, the observe step, and the lifecycle reducer read the same table — no metric is computed two ways.
Author scope by RLSCreators can only touch campaigns/pages/sequences/lists for courses they own — enforced by private.is_course_author(course_id), not app code. Published pages are the only public-readable rows.
Consent & deliverabilityEvery subscriber carries source + consent provenance; every email has a one-click unsubscribe; suppressed states never receive sends.
Mock-first contractsTools depend on provider interfaces, not SDKs. Mock and real implementations are interchangeable behind env-gated factories.

Anti-abuse, consent & deliverability

The product puts email in front of real people and publishes pages publicly — abuse-prone surfaces. Controls are backend-enforced, not UI-only:

  • Send safety = the gate. No list ever receives mail without an explicit human approval recorded in marketing_action; the agent cannot self-authorize a send.
  • Idempotent outbox. Unique (touch_id, subscriber_id) + claimed-status transitions prevent double-sends under repeated scheduler ticks or retries.
  • Consent & unsubscribe. Every subscriber stores source + consent provenance; every email includes a working one-click unsubscribe; unsubscribed/bounced are terminal suppressed states no send can target.
  • Ingest hardening. The service-role route validates the page is published, rate-limits by IP/anon-id, drops malformed payloads, and flags bot traffic; bot/suppressed events are excluded from funnel statistics so the dashboard and the agent reason on clean data.
  • Server-only secrets. OpenAI + Resend + service-role keys never reach the browser; each SDK is imported in exactly one file.

05 — User Experience (Acceptance)

Self-acceptance stories — the core flows that must pass.

Each story is written from the actor’s perspective and tied to concrete on-screen actions. This is core-flow acceptance, not exhaustive QA. Creator stories run in the Marketing area; the lead story runs on the public page.

1Generate the whole kit from the syllabus
User need

I finished my course; I want a complete launch I can review, not a blank marketing tab.

Flow

Open Marketing → click ✦ Generate Kit → watch the agent run.

Expected result

A landing page (7 typed sections), a 4-touch launch sequence, and a behavioral followup all appear as staged drafts, grounded in my outcomes & tone. Nothing is published and nothing has been sent.

2Review and publish the landing page
User need

I want to read the page, tweak it, and put it live deliberately.

Flow

Open the landing draft → Accept the staged sections (or edit slots) → click Publish page… → Approve in the gate.

Expected result

The page goes live at /p/[slug] and is publicly reachable; the action is recorded in the gate ledger. Until I approve, the page is not live.

3A visitor becomes a lead
User need

(Lead) I want the free first lesson; (Creator) I want that person on my list.

Flow

On the public page, enter name + email → click Send me Lesson 1.

Expected result

The lead is added to the creator’s list as lead; page_view + form_submit events land in the stream; the creator sees the count rise on the dashboard. Consent + unsubscribe terms were shown.

4Approve a launch send — the gate in action
User need

I want to email my list, but I must be the one who says “go.”

Flow

Ask the agent to “warm the list,” or click Activate sequence… → review the approval request → Approve & send.

Expected result

Only after approval does the (mock) send fire; email_sent + synthetic open/click events appear and the funnel updates. Deny leaves the list untouched. The agent paused until I decided.

5The agent observes the funnel and proposes the next move
User need

I want the assistant to notice what’s working and suggest improvements, not just generate once.

Flow

Open the Marketing Agent after sends have run → read its observation → accept a proposed followup.

Expected result

The agent cites real funnel numbers (from get_analytics_summary), proposes a targeted followup, and — on my go-ahead — drafts it as a staged, rejectable change. It does not send it on its own.

6Reject a generated asset and get the prior state back
User need

If I don’t like what the agent made, undoing it must be effortless and safe.

Flow

On a staged section or sequence, click Reject.

Expected result

The asset reverts to exactly its before-state (or disappears if newly created), atomically, from the stored snapshot. The reversal persists across reload; nothing partial is left behind.

Definition of done — Marketing Assistant v1

A creator opens Marketing, clicks Generate Kit, and gets a grounded landing page + launch sequence + behavioral followups as staged, rejectable drafts; publishes the page (gated) to a live public route that captures leads into a per-creator list; watches a single event stream populate the dashboard and feed the agent’s observe step; and approves every outward-facing send through one governance gate — with all provider keys server-only, the whole loop running on mock providers, and real Resend + cron swapping in by a single env var with no contract changes.
