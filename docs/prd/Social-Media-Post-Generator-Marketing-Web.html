01 — Purpose & Goals

Turn real course content into platform-ready posts — the creator stays the publisher.

WiseSel creators finish a course and then face a blank social feed. Generic AI copy tools don't know their curriculum; scheduler suites demand OAuth and platform connections before writing a single good post. Phase 1 builds the generation backbone: specific, non-generic, voice-true posts from actual course context — quality over speed (generation takes the time it needs under a hard 3-minute ceiling, streaming drafts as they validate), with zero platform APIs, zero email, zero autonomous posting.

Forward compatibility
The Phase 1 schema and event taxonomy must not foreclose Phases 1.5–4. Reserved fields (tagged FWD throughout — e.g. external_ref, post_type, performance.source) are nullable, unused by Phase 1 UI, and cost nothing. They are not scope creep and not an invitation to build later-phase behavior now.

Goals

1 · Grounded, quality-first generation

Specific, non-generic, platform-appropriate posts from real course context. Generation takes the time it needs (hard 3-minute ceiling); drafts stream in as each validates, so the wait never feels blank.

2 · Batch planning

Up to 5 posts per batch with optional plannedPostAt times — internal planning labels only, never external scheduling.

3 · Funnel-stage awareness

Every post carries a funnelStage (tofu / mofu / bofu); batch generation defaults to a balanced funnel mix.

4 · Creator voice

A per-creator voice profile derived from course and creator context, so output sounds like the creator — not like an AI.

5 · Cursor-like assistant integration

The Marketing Chat Assistant generates, inspects, revises, and manages posts through typed tools with versioned updates that never silently overwrite newer user edits.

6 · Performance logging from day one

Manual metrics or one-tap qualitative rating on posted posts, so closed-loop data starts accumulating now.

7 · Zero external surface

No social platform APIs, no email sending, no autonomous posting — enforced by tests, not just convention.

8 · Production quality

Server-side validated generation, Zod-gated output with one repair pass, RLS-scoped persistence, full test coverage, designed empty/loading/error states.

Non-goals — hard fences

Out of scope · enforced by CI tests, not convention
  • No email sending, drafting-for-send, or email infrastructure of any kind.
  • No auto-posting, auto-scheduling, or "publish" semantics. plannedPostAt is a label, not a trigger.
  • No LinkedIn / Instagram / Facebook / TikTok / X API calls, OAuth, connectors, or platform integrations.
  • No AI image or video generation, image editing, brand templates, auto-resizing, or carousel rendering.
  • No Instagram content for now — Instagram is image-first and WiseSel has no image/video generation yet; it returns when visuals ship (Phase 1.5+). No TikTok/Reels scripts or X/Twitter threads. The platform enum is closed at 2 (LinkedIn, Facebook) for MVP.
  • No content-calendar automation, A/B testing, social analytics ingestion, bot engagement, or recurring background generation.
  • No cron or scheduler infrastructure — nothing in Phase 1 runs on a timer.
  • No approval cards. Every tool here is reversible; blocking approvals for reversible tools was the exact UX failure identified in the autonomy redesign. Do not reintroduce it.

Takeaways

Borrow

Taplio's core lesson: voice is the differentiator. A derived, versioned, creator-tunable voice profile is the single highest-leverage lever against generic AI copy — treat its quality as a first-class test target (§3, §4). Also borrow platform-native register: each platform gets its own style guide, not one caption resized for every network.

Own

Curriculum grounding is unclaimed space. No competitor generates from an actual course structure — real module names, lesson outcomes, target students. WiseSel already holds this context; posts that quote it are unfakeable elsewhere. Funnel staging from day one seeds the Phase 4 closed loop competitors lack.

Defer

The scheduler/OAuth stack is Buffer-class table stakes but heavy: connectors, token refresh, platform policy churn. Phase 1 deliberately wins on draft quality + manual-publish honesty; Phase 3 adds publishing through a unified social API as an irreversible-tier gated action.

03 — Product Approach

One grounded pipeline, one reversible tier, one versioned write path.

Everything — REST routes, assistant tools, UI buttons — funnels into a single service layer running one server-side generation pipeline. All 19 assistant tools run through the existing three-mode autonomy system: the five read/suggest tools execute freely, and all fourteen write tools are classified reversible, so agent actions auto-commit with a passive revert log and no blocking approval card appears anywhere in the feature. Every content write goes through one versioned-update statement; a stale version is a 409, never a silent overwrite.

Decisions taken (alternatives considered and rejected)
  • One structured-output call per batch, not N per-post calls — the funnel mix and ordering are reasoned together, it is cheaper, and it counts as one call against the platform-wide two-concurrent-model-call ceiling. No latency target: generation is quality-first under a hard 3-minute timeout (SOCIAL_GENERATION_TIMEOUT_MS).
  • Reversible auto-commit + revert log, not approval cards — approval cards on reversible tools were the exact failure the autonomy redesign removed; reintroducing them here is forbidden.
  • Manual publish, not platform connectors — publishing is deferred to Phase 3 where it becomes an irreversible-tier gated action; Phase 1 stays honest ("you post this yourself").
  • Soft delete only — deleteSocialPost archives with deletedAt; hard purge is a later-phase retention job.

Post lifecycle

draft→ ready→ planned→ posted_manual→ archived

Batch planning & funnel mix

Batch rules

  • Hard max 5 posts per batch, enforced in four layers on purpose (cost + safety control): Zod input schema, service layer, DB check (batch_order between 1 and 5), and UI.
  • A batch is a first-class row (social_post_batches) so the queue can group, retry, and audit per batch. Persistence is transactional — all posts commit or none.
  • Timing presets compute suggested times client-side in the creator's timezone: same day (≥2h apart, next full hour) · spread this week (one weekday-ish slot each, 9:00–11:00 local, never two per day) · spread 2 weeks (~every 2–3 days) · custom · none.
  • Suggested times carry ±20min jitter so followed plans don't look robotic.

Creator voice profile — the anti-generic lever

What it is

One active, versioned row per creator (creator_voice_profiles): a 2–4 sentence style summary, register, sentence-length tendency, emoji tolerance, signature moves ("opens with a question", "numbered mini-lists"), creator-editable banned phrases, and up to 3 creator-pasted sample excerpts.

How it behaves

Derived on first generation by a small-tier call over course descriptions + lesson intros + delivery_profile + any pasted samples. Viewable and regenerable from a settings sheet inside Social Posts. Creator edits persist and set source='creator_edited' — regeneration then requires confirmation. Injected into every generation and revision prompt. Test target: two distinct creator fixtures must produce distinguishably different posts.

Governance — how agent writes behave

Autonomy modeBehavior (reads execute freely in every mode; all writes are reversible)
assisted / autoAuto-commit with a revert-log entry. Quiet, dismissible — never a blocking card.
manualThe assistant proposes; the creator clicks apply. Existing mechanism — nothing new built for governance.
The versioned-write rule (the only legal update path)
update social_posts set …, version = version + 1, updated_at = now()
  where id = $1 and version = $2 and deleted_at is null returning *;

Zero rows returned ⇒ 409 to the caller. No other UPDATE may touch content fields — all writes funnel through one repository function, and a lint/test greps for strays. On 409 the assistant re-reads, re-applies intent against the fresh content, and says so. It never force-writes.

Business KPIs

The four success criteria in §01 are the Phase 1 KPI set (save-without-rewrite ≥80%, the 3-minute generation ceiling, zero silent overwrites, 100% lint pass). Current analytics for this surface do not exist — all current values are N/A; instrumentation ships with the feature (every copy/download/edit/status event fires from day one), so baselines accumulate immediately. Finer-grained metrics wait until these four are measured.

FWD post_type ('text' now; 'clip'/'carousel' in 1.5), external_ref (Phase 3 publish state). Indexes: (creator, status) and (creator, planned_post_at) where not deleted; (batch_id).

creator_voice_profiles

One active row per creator: profile jsonb (summary, register, sentenceLength, emojiTolerance, signatureMoves[], bannedPhrases[], sampleExcerpts[]), source (derived | creator_edited), version.

  • RLS: all three tables scoped creator_id = auth.uid() for select/insert/update; no delete policy (soft delete only). The generation service uses the service role. Follows the repo's existing RLS test pattern.
  • Zod is the single source of truth — enums (Platform, Goal, FunnelStage, Tone, PostStatus), GeneratedPostSchema (per-platform max via superRefine against PLATFORM_LIMITS), GeneratedPostBatchSchema (1–5), GenerateRequestSchema (source-id consistency), SocialPostSchema row mirror, PostPerformanceSchema. Inferred TS types everywhere; never hand-written. The Platform enum is linkedin | facebook — deliberately closed at 2; Instagram joins when image/video generation ships (Phase 1.5+).
  • Events extend the shared discriminated union — client SDK and server ingest in the same change: social_post.batch_generated · created · updated (fieldsChanged, byAgent) · revised_by_agent · status_changed · copied · downloaded · image_attached / image_removed · performance_logged · generation_failed (stage: model|zod|repair|lint) · voice_profile.derived / edited. Standard envelope; publication-version field null (marketing events, not course-consumption events — documented).

Cross-cutting rules

  • One service layer (services/socialPosts/*) behind REST and tools; reuse the existing course context retriever — do not fork a second one.
  • Model keys server-only; cache-stable prefix; mid-tier generation / small-tier revisions; existing semaphore honors the 2-concurrent ceiling.
  • Rate limits (config, not constants): SOCIAL_MAX_BATCHES_PER_DAY = 20, SOCIAL_MAX_REVISIONS_PER_DAY = 100 → typed 429 the UI renders kindly.
  • Generation timeout: SOCIAL_GENERATION_TIMEOUT_MS = 180,000 (config). No latency target — quality first; on timeout a typed error returns, nothing persists, and the batch parameters are kept for retry.
  • Idempotency: POST /generate accepts an Idempotency-Key; duplicates return the original batch.
  • Observability: ai_metadata on batch + post rows; generation_failed events on every failure path.
  • No cron, no webhooks, no outbound calls to any social host (asserted by an allowlist test).

Safety — prompt rules + deterministic lint

Prompt-level (stable system prefix)

No fake testimonials · no fake student outcomes · no misleading income/result claims · no artificial scarcity/urgency unless verbatim in creator-supplied context · no hashtag stuffing · no platform-policy circumvention advice · no engagement-bait formulas ("comment YES if…") unless tone=casual AND the voice profile shows it · every post includes a suggestedImageIdea that is shootable/ownable by the creator ("photo of your palette mid-lesson", never "generate an image of…") · manual publishing is the model and copy must never contradict it.

Deterministic lint (code, post-Zod, table-driven tests)

Regex/heuristic flags:

  • Currency + earnings verbs ("made $", "earn $X")
  • "Students got/achieved <number>" patterns
  • "Only X spots/hours left" absent from source context
  • Testimonial quote patterns ("…" — Name) not in source context
  • Hashtags over platform max · ALL-CAPS ratio > 30%

Flag ⇒ one targeted repair instruction (within the single repair budget) or drop the post with a surfaced reason. Creator-provided context whitelists its own claims (false-positive escape hatch).

Image handling

Upload only — never generated. jpeg/png/webp ≤ 10MB to bucket social-post-images at creators/{creatorId}/social/{postId}/{uuid}.{ext}, RLS-scoped, signed URLs with short TTL. Flow: signed upload → finalize endpoint → server validates content-type + size + dimensions → soft warning (never a block) if far from platform norms (LinkedIn 1200×627, Facebook 1200×630). Alt text prominent; the assistant may draft it on request but never auto-writes it. Remove detaches the reference; the object is retained (revert-friendly) until a later retention purge.

05 — User Experience (Acceptance)

Self-acceptance stories — the core flows that must pass.

Five stories at core-flow level, each grounded in visible prototype actions (Screens 1–8). Exhaustive QA lives in the milestone test plans (§04); the full acceptance list in the source spec (19 criteria) maps onto the automated suites.

Story 1Generate a balanced batch from a real course
User need"I finished Watercolor Foundations. Give me a week of posts that sound like me and actually reference my course."
FlowOpen Marketing → Social Posts (Screen 1) → source = Course · Watercolor Foundations → platform LinkedIn → goal Educational/value (stage chip shows tofu) → tone Friendly → count 5 with Balanced funnel mix ON → timing Spread across this week → click Generate 5 drafts.
Expected resultDrafts stream into the queue one by one (Screen 2), never a monolithic spinner. Exactly 5 posts: 3 tofu, 1 mofu, 1 bofu, ordered value-first with bofu planned last. Bodies quote real module/lesson names ("Wet-on-wet techniques"). Each card carries platform, stage chip, CTA, hashtags within LinkedIn's 0–5 range, a shootable image idea, a jittered weekday-morning planned time, status=draft. No 6th post is possible from UI, Zod, service, or DB.
Story 2Edit safely — versions protect every write
User need"I tweak drafts myself and also let the assistant polish them — my edits must never get silently clobbered."
FlowOpen a draft in the post editor (Screen 4) → edit the hook line manually → Save → then in Marketing Chat ask "make this punchier" while the editor still holds the older version → assistant runs reviseSocialPost with its expectedVersion.
Expected resultThe manual save increments the version. The assistant's stale write hits the 409 path: it re-reads the fresh body, re-applies "punchier" on top of the manual edit, and says it did so. The editor shows the non-blocking toast "Updated elsewhere — refreshed to latest" (Screen 7). At no point is either edit lost; History shows both entries.
Story 3Attach an image, export, and mark posted manually
User need"I want the finished post plus my own photo in my hands, then I'll post it on LinkedIn myself."
FlowIn the editor (Screen 4): upload palette-midwash.jpg → preview renders with a dimension check → fill the alt-text field → Copy post → Download .md → post natively on LinkedIn → back in WiseSel set status to Posted manually.
Expected resultCopy puts body + CTA on the clipboard with a success toast; the .md download carries front-matter (platform, stage, plannedPostAt, status — never anything publish-like) plus body, CTA, hashtags; both fire their export events. The status change stamps postedManuallyAt and the queue card's action becomes Log performance (Screen 3). The manual-publish notice was visible throughout; a failed upload would have left the post intact.
Story 4Log how the post did — in one tap if that's all I know
User need"I don't track spreadsheets. Let me record 'that one did well' and move on."
FlowOn a posted-manual card, click Log performance (Screen 6) → tap the qualitative rating OK (optionally add impressions/likes/comments) → Save log.
Expected resultThe log saves with either signal alone (one-tap qualitative is sufficient), persists to the performance JSONB with source:'manual' and loggedAt, fires social_post.performance_logged, and shows on the post. Nothing else changes — no analysis, no automation; the data simply starts accumulating for Phase 4.
Story 5Do it all conversationally — and stay revertible
User need"I'd rather just tell the assistant what I want than click through controls."
FlowIn Marketing Chat (Screen 8): "Generate 5 posts for Watercolor Foundations spread over next week, mostly value posts" → then "Make #3 punchier and rewrite #4 for Facebook" → open Recent changes and revert the revision of #3.
Expected resultThe assistant calls generateSocialPostDrafts, summarizes the batch, then reads #3 (getSocialPost) before reviseSocialPost + rewriteForPlatform — each write auto-commits quietly with a revert-log entry; no approval card ever appears. The Facebook rewrite is a new post; the LinkedIn original is untouched. Each revision comes with a one-to-two-sentence "why this may perform better". The revert restores #3's previous version byte-for-byte. If asked to auto-post, the assistant explains the manual-publish model in one sentence and never claims anything was published.
