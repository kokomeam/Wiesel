01 — Purpose & Goals

Turn the studio’s deterministic mock into a real, trustworthy authoring partner.

CourseGen’s thesis is “Cursor for courses”: the studio is creator-economy table-stakes; the AI layer is the moat. Today every AI action is a deterministic mock behind a single function (requestAIPatches). This milestone ships the first real agent — one versatile content agent that authors lessons, slide decks, knowledge checks, and practice work by calling tools, and earns trust by showing its changes before they’re accepted.

The problem

  • Authoring is slow and blank-page-hard. Creators have expertise but not time; the studio gives them a canvas, not a collaborator.
  • The current “AI” can’t actually help. The mock returns canned patches; it can’t reason about a course, write grounded content, or hold a conversation.
  • Generative tools feel unsafe. Creators won’t let an AI silently rewrite their course. They need to see, understand, and approve every change.

Goals

Real, grounded generation

An agent that writes lessons/slides/checks grounded in the course’s outcomes, level, and teaching tone — quality over speed.

Trust by review

Every mutation is staged, highlighted in the editor, and individually accept/rejectable. Nothing is irreversible.

Future-proof architecture

Provider-agnostic, ops-layer-only writes, DB-coordinated — so specialist agents and an orchestrator slot in later with no rework.

Non-goals (this milestone)

  • No image generation — leave a clean seam only.
  • No multi-agent orchestration — one content agent; specialists/orchestrator come later via shared DB state, not agent-to-agent messaging.
  • No gradebook. Assessments are low-stakes knowledge checks & practice — see the structural guarantee in §4.
  • No marketplace/Stripe/analytics changes.

Hard non-negotiables (enforced by design)
  • OpenAI key & every model call are server-side only — the SDK is imported in exactly one file.
  • All content mutations flow through the existing operations layer (the validated patch pipeline the studio UI already uses). The agent has no private write path.
  • Assessments are low-stakes: no scores, passing thresholds, time limits, attempt caps, difficulty tags, points, or due dates — the schemas must not even contain those fields.
  • RLS & migrations require human sign-off before applying.

02 — Competitive Analysis

Borrow the agentic-review pattern from code tools; the slide aesthetic from deck AIs; reject the LMS gradebook.

No competitor screenshots were supplied, so this is reasoned from well-known product patterns and marked as inference. We compare only decision-relevant dimensions: entry point, the agentic loop, how changes are reviewed, grounding, and trust cues.

ProductCore pattern (inferred)What to borrowWhat to avoid
Cursor / Copilot agents
code editor		 Chat docked beside the editor; an agent runs a tool loop over the workspace, proposes edits as a reviewable diff, you Accept/Reject per hunk. The whole interaction model: docked chat + agentic tool loop + staged, per-unit accept/reject. This is our north star. Code-diff density & jargon. Course creators aren’t engineers — show block-level change cards, not text diffs.
Gamma / Slides AI
deck gen		 Prompt → a whole deck appears; light per-slide regeneration; strong default aesthetics. Minimal, professional slides — few words per slide, strong layouts, no walls of text. Bake this into the slide-writer tool. One-shot “magic” with no review and no grounding in a broader curriculum. Opaque = untrustworthy for paid courses.
Notion AI / Docs AI
inline writing		 Inline “ask AI” that rewrites the current selection; conversational follow-ups. Conversational follow-ups & next-step suggestions grounded in current state; the “Magic Wand” iterative feel. Selection-only scope. Our agent must operate across a lesson (and neighbors) via read tools, not just one block.
LMS course-AI (Coursebox, LearnWorlds, etc.)
edtech		 AI generates quizzes/courses; assessments are graded — scores, passing %, attempts, timers. Breadth of artifacts (lesson + check + practice) from one brief. The gradebook. Our product is explicitly low-stakes; right/wrong is an analytics signal, never a grade. We strip those fields entirely.
Net takeaways

1. Adopt Cursor’s docked-agent + staged-review model, but express changes as block-level cards, not diffs. 2. Bake Gamma-grade slide minimalism into the writer tools’ schema & prompt. 3. Ground every artifact in the course plan (outcomes, level, tone) — the thing one-shot tools skip. 4. Structurally exclude gradebook semantics that LMS tools assume.

03 — Product Approach

A docked chat agent that writes through the studio’s own tools and stages every change.

Approaches considered

A · Inline one-shot command

Extend the existing command bar to call a real model and apply patches directly.

Rejected: no review, no conversation, no multi-lesson reasoning. Doesn’t build the moat.

B · Docked agent + tool loop + change-set review ✓

Cursor-style chat panel; a server-side agentic loop calls ops-layer tools; mutations persist but stage as a reviewable change-set.

Recommended: trustworthy, conversational, extensible to specialists/orchestrator.

C · Background “generate whole course” batch

A wizard that generates an entire course unattended.

Deferred: powerful but opaque; no per-change trust. Becomes easy after B (it’s the same loop, unattended).

Why “apply then stage,” not “stage then apply”: the DB stays the single source of truth, so on later loop steps the agent can read its own pending edits and build on them. Rejecting replays the stored before-snapshots through the same patch pipeline.

Key business rules

Server-only model accessThe agent runs entirely in route handlers. The browser never sees the key, the provider SDK, or raw model traffic.
Ops-layer writes onlyEvery change is a validated CoursePatch applied by the same pure reducer the studio UI uses. Invalid output from the model is rejected, never persisted.
Author scopeThe agent can only read/write courses the signed-in author owns (enforced by RLS, not app code). Learners never see agent edits or pending changes.
Low-stakes assessmentsQuiz/homework content cannot carry scores, passing marks, timers, attempts, difficulty, points, or due dates — guaranteed at the schema level.
Provider-agnosticConversation history lives in our DB and is replayed each turn; we never depend on provider-side session state.

The agent’s tools = the operations layer

Every tool wraps existing patch/command logic — there is no separate write path. Schemas are generated from the studio’s Zod definitions (single source of truth) and sent to the model as strict function schemas.

KindToolPurpose
readget_course_contextCourse title, description, audience, level, and plan (outcomes, prerequisites, teaching style/tone).
list_modules / list_lessonsNavigate structure for multi-lesson jobs.
get_lessonA lesson with its blocks — so the agent aligns with what’s taught and avoids duplication.
get_blockFull content of one block for targeted edits.
(neighbors)Agent pulls siblings on demand rather than stuffing everything into context.
structuralcreate_module / create_lessonExtend the course outline.
create_block / delete_blockAdd or remove a block of a given type.
reorder_blocksResequence blocks within a lesson.
update_blockTitle / structural metadata on an existing block.
writerwrite_slide_deckAuthor a deck — minimal, professional slides, few words each, strong layouts.
write_quizA small, consistent set of knowledge-check questions, each with instant feedback + a short explanation. No scoring fields.
write_homeworkA practice exercise — instructions + optional qualitative rubric/solution. No points/due dates.
write_lecture_textClear, well-structured prose grounded in outcomes & tone.

Writer tools build a complete, schema-valid block and commit it via a single additive patch (SET_BLOCK_CONTENT) validated against the block’s discriminated-union schema — the clean primitive for whole-block generation, fully inside the existing pipeline.

Low-stakes guarantee (structural)

To honor “the schemas must not even contain those fields,” these fields are removed from the quiz/homework content model, their patches, and the studio UI that edited them. Legacy stored data is tolerated (unknown keys are dropped on read), so no data migration is needed.

Removed fieldLived onWhy it’s gone
passingScore timeLimitMinutes attemptsAllowedQuiz settingsNo passing thresholds, timers, or attempt caps — checks are formative.
difficulty pointsQuiz questionNo difficulty tags or scoring; right/wrong is analytics only.
dueAt pointsHomeworkNo due dates or grades — homework is practice.
pointsRubric levelRubric levels become qualitative (label + description), not scored bands.

Kept: question prompt, choices/answer, and a short explanation (instant feedback); homework instructions, exercises, optional solution & qualitative rubric; links to learning outcomes.

Data model (new) & access control

Four new tables, each carrying a denormalized course_id so row-level security can authorize in one hop (matching the existing schema convention). All are author-scoped; nothing is learner-visible.

TableShapeAccess
conversationsscoped to a course (+ optional lesson); a titled thread.RLS: author full CRUD where they own the course; no public/learner read. Migration & policies ship for human review and are applied only after sign-off; types regenerated after.
messagesrole + content (incl. tool calls/outputs) as JSON; replayed each turn.
change_setsone per agent turn; status pending → accepted / rejected; summary.
change_set_itemsper-block op (create/update/delete) + before/after snapshots.

Grounding & cost control

  • Context assembly: a stable system prompt (role + content-quality + low-stakes rules) and the course “prefix” (title/description/audience/level + teaching style/tone) lead every turn; the target lesson and sibling summaries follow. Multi-lesson jobs pull neighbors via read tools instead of bloating context.
  • Prompt caching: the stable prefix is placed first so repeated turns in a tool loop hit the provider’s automatic cache.
  • Bounds: a per-turn tool-call cap with a user “continue?” checkpoint; a capped output budget; the default model is configurable via env.

05 — User Experience (Acceptance)

Self-acceptance stories — the core flows that must pass.

Each story is written from the creator’s perspective and tied to concrete on-screen actions in the studio. This is core-flow acceptance, not exhaustive QA.

1Generate a deck and a knowledge check from one prompt
User need

I want a first draft for this lesson without starting from a blank page.

Flow

Open the lesson → open the agent panel → type “write a 5-slide intro deck and a 4-question knowledge check” → send.

Expected result

The agent streams its work; a slide-deck block and a quiz block appear with amber “pending” rings; the chat summarizes both. The quiz has 4 questions, each with an explanation and no scoring fields.

2Review and accept a change
User need

I want to read what the agent wrote and keep what’s good.

Flow

Click View on the pending slide-deck block → read the slides → click Accept (or Accept all in the review bar).

Expected result

The amber highlight clears, the block becomes a normal part of the lesson, and the change persists — it’s still there after a page reload.

3Reject a change and get the prior state back
User need

If I don’t like a change, undoing it must be effortless and safe.

Flow

On a pending block, click Reject.

Expected result

The block reverts to exactly its before-state (or disappears if the agent created it). The reversal persists across reload; the chat notes it was rejected.

4Have a grounded follow-up conversation
User need

I want to refine the result by talking, and pick up where I left off.

Flow

After accepting the deck, type “make slide 3 simpler and add an example” → send. Later, reload the page and reopen the panel.

Expected result

The agent edits the right slide with awareness of current lesson state and suggests a next step. After reload, the full conversation is still there.

5Flesh out several lessons in one job
User need

I want the agent to handle a bigger task across multiple lessons.

Flow

Type “flesh out the next three lessons in this module with lecture text and a knowledge check each” → send → if prompted, click Continue.

Expected result

The agent navigates the module via its read tools and adds grounded, non-duplicative content to each lesson as pending changes; large jobs pause at a checkpoint the creator can continue.

Definition of done — milestone 1

A creator opens a lesson, asks the docked agent for a deck + knowledge check, watches it stream and apply changes through the ops layer, sees them highlighted & summarized, then accepts or rejects — with the OpenAI key living only on the server, all content valid against the (now low-stakes) schemas, and no schema/RLS change applied without human sign-off.
